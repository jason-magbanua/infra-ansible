# infra-ansible

Ansible playbooks and roles for configuring infrastructure provisioned by Terraform on Proxmox VE. Terraform handles provisioning; Ansible handles everything after.

---

## Structure

```
ansible/
├── ansible.cfg
├── collections/
│   └── requirements.yml        # community.general
├── inventory/
│   └── lab/
│       ├── hosts               # generated by Terraform (gitignored)
│       └── hosts.example       # reference template
├── playbooks/
│   ├── site.yml                # Full WordPress VM pipeline (lemp → wordpress → tuning)
│   ├── lemp.yml                # Install Nginx, MariaDB, PHP on VM
│   ├── wordpress.yml           # Deploy WordPress on VM
│   ├── performance-tuning.yml  # Adaptive Nginx + PHP-FPM tuning
│   ├── setup-zfs.yml           # ZFS mirror pool setup on VM host
│   ├── setup-lxc.yml           # LXD setup on VM host (storage, network, image)
│   ├── install-wordpress-on-container.yml  # WordPress inside an LXD container
│   └── templates/
│       ├── nginx-adaptive.conf.j2
│       ├── nginx-wordpress.conf.j2
│       ├── php-fpm-www.conf.j2
│       └── wp-config.php.j2
└── roles/
    ├── mysql_tuning/           # MariaDB slow log + tuning config
    ├── nginx_cache/            # FastCGI cache dir + vhost deploy/enable
    ├── php_tuning/             # OPcache configuration
    └── redis/                  # Redis + php-redis extension
```

## Requirements

- Ansible >= 2.15
- Python >= 3.10
- `community.general` collection (see [collections/requirements.yml](collections/requirements.yml))

Install dependencies:

```bash
python3 -m venv venv
source venv/bin/activate
pip install ansible

ansible-galaxy collection install -r collections/requirements.yml
```

## Inventory

The inventory at `inventory/lab/hosts` is **generated automatically** by `terraform apply` — do not edit it by hand. Use `inventory/lab/hosts.example` as a reference for the expected format.

Host groups:

| Group                | Contents                        |
|----------------------|---------------------------------|
| `proxmox`            | Proxmox hypervisor node         |
| `proxmox_vms`        | Proxmox VMs (docker-host, wp-*)  |
| `proxmox_containers` | LXC containers (root user)      |

## Usage

### WordPress VM pipeline

Full stack deploy (LEMP → WordPress → performance tuning):

```bash
ansible-playbook playbooks/site.yml
```

Or run stages individually:

```bash
ansible-playbook playbooks/lemp.yml
ansible-playbook playbooks/wordpress.yml
ansible-playbook playbooks/performance-tuning.yml
```

### WordPress in an LXD container

Provision a new WordPress site inside an LXD container on `wp-host1`:

```bash
ansible-playbook playbooks/install-wordpress-on-container.yml \
  -e container_name=mysite \
  -e domain=mysite.example.com
```

This is idempotent — re-running against an existing container skips install steps and only updates `wp-config.php` and resource limits.

#### Default container resources

| Resource | Default |
|----------|---------|
| CPU      | 1 core  |
| Memory   | 512 MB  |
| Disk     | 5 GB    |

Override at runtime without editing the file:

```bash
ansible-playbook playbooks/install-wordpress-on-container.yml \
  -e container_name=mysite \
  -e domain=mysite.example.com \
  -e container_cpu=2 \
  -e container_memory=1GB \
  -e container_disk=10GB
```

#### DB credentials

On a fresh install, credentials are auto-generated and saved to `/opt/infra/secrets/<container_name>-db.yml` (mode `0600`). On re-runs, credentials are loaded from that file and used to re-render `wp-config.php`.

A deployment summary is printed at the end of every run.

#### HTTPS behind a reverse proxy

`wp-config.php` sets `WP_HOME`/`WP_SITEURL` to `https://{{ domain }}` and trusts the `X-Forwarded-Proto` header from the upstream proxy (e.g. Nginx Proxy Manager). This prevents mixed-content errors when SSL is terminated upstream.

### ZFS + LXD host setup

Run once after a new VM host is provisioned:

```bash
ansible-playbook playbooks/setup-zfs.yml
ansible-playbook playbooks/setup-lxc.yml
```

Pass `--extra-vars debug_output=true` to `setup-lxc.yml` to print storage/network/image state at the end.

#### Connecting the infra server to an LXD remote

Before Ansible can manage LXD containers on a VM host, the infra server's `lxc` client must be trusted by that host's LXD daemon.

LXD does not listen on the network by default. Enable it first **on the VM host**:

```bash
lxc config set core.https_address :8443
```

Then generate a one-time trust token **on the VM host**:

```bash
lxc config trust add --name infra-server
```

Finally, add the remote **on the infra server** using the printed token:

```bash
lxc remote add wp-host1 https://<vm-host-ip>:8443 --token <token>
```

Verify the connection:

```bash
lxc remote list
lxc ls wp-host1:
```

The token is single-use and short-lived — run all three steps in the same session.

## Roles

| Role           | Purpose                                              | Key variable       |
|----------------|------------------------------------------------------|--------------------|
| `mysql_tuning` | Deploys MariaDB tuning config + slow query log       | —                  |
| `nginx_cache`  | Creates FastCGI cache dir, deploys and enables vhost | `wordpress_domain` |
| `php_tuning`   | Configures OPcache via ini drop-in                   | `php_version`      |
| `redis`        | Installs Redis + versioned php-redis extension       | `php_version`      |

Default `php_version` is `8.3` (set in `roles/php_tuning/defaults/main.yml`).

## Security

- Sensitive data must never be committed — use `ansible-vault` for secrets
- `host_key_checking = False` in `ansible.cfg` is intentional for the homelab; enable it and pre-populate `known_hosts` for stricter environments
- LXC containers run as `root` — this is expected for Proxmox unprivileged containers
